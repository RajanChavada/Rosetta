# Config-Driven Scaffolding – Overview

## Goal

Allow projects to define Rosetta behavior via configuration files, minimizing or eliminating interactive prompts.

Key themes:
- `defaultPreset`, `defaultIdes`.
- Default context values.
- Per-project skill & preset sources.
- Post-scaffold hooks.

## Config File

Primary file: `.rosetta.json` at repo root.

Example:

```json
{
  "defaultPreset": "agentic-starter",
  "defaultIdes": ["VSCode / Claude Code", "Cursor"],
  "autoContext": {
    "enabled": true,
    "projectName": "AtlasPay",
    "description": "Multi-tenant payments platform",
    "projectType": "API / backend service",
    "domainTags": ["Financial"],
    "riskLevel": "High (Critical/Financial/Healthcare)"
  },
  "skills": {
    "alwaysInclude": ["testing-full-pyramid", "node-express-postgres"]
  },
  "skillSources": [
    ".rosetta/skills",
    "company-skills"
  ],
  "postScaffoldHooks": [
    "npm install",
    "npm run lint"
  ]
}
```

## Modes

1. **Interactive default** (no config).
2. **Config-augmented**: Config pre-fills answers, CLI can still ask to confirm.
3. **Config-only / non-interactive**: `rosetta scaffold --non-interactive` uses `.rosetta.json` exclusively. Fail fast if required fields are missing.
